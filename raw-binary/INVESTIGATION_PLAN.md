# План исследования крэша после coex_wifi_release

## Что видно из лога

1. **Последовательность до падения:**  
   `coex_enable` → `coex_wifi_request event=36 latency=0 duration=1000000` → `coex_wifi_release event=36` → **Exception: pc: nil, code: 2, mcause: 0x00000002 (Illegal Instruction)**.

2. **Интерпретация:**  
   Управление возвращается из нашей заглушки `coex_wifi_release` в код блоба. Следующая исполняемая инструкция — по адресу **0** → вызов/переход по нулевому указателю (например, callback или функция в таблице не инициализированы).

3. **Полезные адреса из лога:**  
   - Колбэки coexistence, зарегистрированные нами: `coex_register_start_cb cb=0x420151b4`, `coex_schm_register_cb type=0 cb=0x420151f4`.  
   - Эти адреса в образе приложения (0x42...), их можно искать в `firmware.elf.dis`.

## Шаг 1: Код блоба сразу после вызова coex_wifi_release

**Цель:** Найти в блобе место: «вызов _coex_wifi_release через osi → следующая инструкция после return». Там, скорее всего, загрузка и вызов ещё одного указателя (callback/функция), который у нас равен 0.

**Как искать:**

- В `firmware.elf.dis` код блобов и приложения смешан; адреса 0x4038xxxx — типично IRAM (в т.ч. код из libnet80211, libpp и т.д.).
- Поскольку в `.a` дизассемблер даёт смещения 0, в связанном образе нужно искать по смыслу:
  - вызов по указателю (например, `jalr` по регистру, загруженному из памяти);
  - сразу после этого — ещё одна загрузка из памяти и `jalr`/`jr` по регистру.

**Где искать:**

- `raw-binary/libs/esp32c3/libnet80211.dis` — символы `esp_wifi_internal_on_coex_start`, `esp_wifi_internal_on_coex_schm_phase`, `wifi_on_coex_start_process`, `wifi_on_coex_schm_phase_process`: они вызывают колбэки через указатели (в дизассе там есть `lw` + `jalr`/`jr` по нулевому смещению в объектах).
- `raw-binary/libs/esp32c3/libpp.dis` — `pm_coex_schm_process`, `pm_on_coex_start` и т.п.: аналогично, вызов по указателю после какой-то операции.

Сопоставить эти фрагменты с `firmware.elf.dis` по адресам (через символы или по характерным последовательностям инструкций), затем смотреть код **сразу после** возврата из вызова по указателю (наш `coex_wifi_release` вызывается так же).

## Шаг 2: Что делают наши колбэки 0x420151b4 и 0x420151f4

**Цель:** Убедиться, что это действительно `coex_register_start_cb` и `coex_schm_register_cb`, и не делают ли они чего-то, что может обнулить указатель в структуре блоба.

**Действия:**

- Проверить по коду: это короткие обёртки (например, сразу `ret` или вызов одной функции и возврат).
- Сверить с реализацией в `osi.c` (какие функции подставлены в `_coex_register_start_cb` и `_coex_schm_register_cb`).

## Шаг 3: Кто вызывает coex_wifi_request(36) и coex_wifi_release(36)

**Цель:** Найти один и тот же путь в коде, где делаются request и release для event=36. По этому пути будет видно, что именно выполняется после release (и какой указатель там может быть 0).

**Действия:**

- В дизассе блобов искать константу **36** (0x24) как аргумент: например загрузка `li a0, 36` или аналог перед вызовом. Искать в `libnet80211.dis` и `libpp.dis`.
- Найти функцию, которая:
  - вызывает «request» с event=36;
  - потом что-то делает (или вызывает release);
  - после return из release что-то загружает из памяти и делает `jalr`/`jr`.
- Этот фрагмент после return и есть кандидат на переход по NULL.

## Результат (найдена причина)

В `wifi_reset_mac()` после `coex_wifi_release(0x24)` код делает:
- загрузку по адресу `g_wdev_last_desc_reset_ptr` (0x3ff1ee40);
- запись байта 1 по полученному указателю: `*(*g_wdev_last_desc_reset_ptr) = 1`.

Символ `g_wdev_last_desc_reset_ptr` в линкере задан как адрес 0x3ff1ee40; по этому адресу должен лежать **указатель на байт**. Если туда ничего не записано (0), происходит запись по адресу 0 → сбой.

**Исправление:** в `radio.c` перед вызовом `esp_wifi_init_internal()` инициализировать ячейку по 0x3ff1ee40 адресом нашей переменной `g_wdev_last_desc_reset_byte`, чтобы блоб писал флаг в неё, а не по NULL.

**Проверка по raw-binary/libs:** В `libnet80211.dis` в секции `.text.wifi_reset_mac` (строки ~33548–33584) последовательность такая: вызов по osi+408 (request), по osi+244, по osi+412 (release), затем `lui a5,0x0` + `lw a5,0(a5)` (загрузка по адресу с релокацией — это g_wdev_last_desc_reset_ptr), `li a4,1`, `sb a4,0(a5)`. То есть блоб только **читает** ячейку по адресу g_wdev_last_desc_reset_ptr и пишет 1 по полученному указателю; **записи в саму ячейку 0x3ff1ee40** в этом .o нет. Поиск по всем `raw-binary/libs/esp32c3/*.dis` по подстрокам `g_wdev_last_desc_reset`, `3ff1ee40`, `last_desc_reset` — совпадений нет. Значит, ни один объект в наших блобах не инициализирует значение по 0x3ff1ee40.

**Откуда данные по задумке:** В дизассе `wdev_data_init()` (libpp) заполняются только указатели в диапазоне 0x3fcdfxxx (pTxRx, wDevCtrl_ptr, wDevMacSleep_ptr, g_lmac_cnt_ptr, …). Записи по 0x3ff1ee40 нет — блоб этот указатель нигде не выставляет. Значит, в полном IDF его либо выставляет другой объект/библиотека, либо линкер кладёт туда начальное значение; в нашей сборке такого источника нет, поэтому явная инициализация в `espradio_wifi_init()` — не костыль, а необходимое заполнение контракта блоба. ROM-функцию блоб тут не дергает: он только загружает указатель из 0x3ff1ee40 и пишет байт 1 по этому адресу.

**ROM:** В линкере есть только присвоение адреса: `g_wdev_last_desc_reset_ptr = 0x3ff1ee40` (ячейка RAM). Функции ROM в ld — по адресам 0x4000xxxx (spiflash, cache, phy, rom_pp, rom_net80211, rom_coexist и т.д.); аналога `btdm_controller_rom_data_init` для Wi‑Fi/wdev нет, ни одна ROM-функция не записывает значение по 0x3ff1ee40.

**Bootloader:** Адрес 0x3ff1ee40 не относится к образу bootloader’а. Второй этап загрузчика — отдельный бинарник со своими секциями; он не линкует символ `g_wdev_last_desc_reset_ptr`. В нашем линкере DRAM приложения — 0x3FC80000..0x3FCCE710; 0x3ff1ee40 лежит вне этого диапазона (зона WiFi/ROM). Ни в прошивке приложения, ни в bootloader’е нет сегмента, который бы *загружался* по 0x3ff1ee40 — это только адрес ячейки, куда блоб ожидает указатель. Bootloader при старте приложения не пишет по этому адресу.

## Заметка про event=36

В `esp_wifi_types_generic.h` нумерация `WIFI_EVENT_*` может не совпадать с числом 36 (зависит от платформы и конфига). Имеет смысл один раз сопоставить константу 36 с конкретным событием в твоей сборке (по enum и комментариям), чтобы понимать, на каком сценарии (scan, home channel, и т.д.) воспроизводится крэш.

## Крэш сразу после espradio_queue_recv return (pc: nil)

**Лог:** `espradio_queue_recv got cmd 6` → `espradio_queue_recv return` → **Exception: pc: nil**.

**Смысл:** Блоб вызывает наш `_queue_recv` (слот 116 в `wifi_osi_funcs_t`). После возврата выполнение идёт в код блоба; следующая инструкция даёт переход по адресу 0 → либо вызов по нулевому указателю (ещё один слот таблицы OSI), либо испорченный адрес возврата (stack).

### Как искать точное место в build/firmware.elf.dis

1. **Слот queue_recv:** в `wifi_os_adapter.h` у `_queue_recv` смещение **116** (0x74) от начала структуры. В дизассе блоба это даёт инструкцию вида `lw a5, 116(a5)` или `lw a5, 116(s0)` после загрузки указателя на таблицу из `g_osi_funcs_p`.

2. **Паттерн вызова queue_recv в связанном образе:**
   - Загрузка таблицы: `lw a5, -1708(s0)` или аналог по `g_osi_funcs_p` (0x3fcdf954).
   - Загрузка слота 116: `lw a5, 116(a5)` — в objdump может отображаться как `0747a783` или `74(a5)`.
   - Подготовка аргументов (queue, item, block_time) в a0, a1, a2.
   - Вызов: `jalr a5` (код 9782).

3. **Что смотреть после этого jalr:**  
   Следующие 5–15 инструкций после `jalr a5` — это код, выполняемый после возврата из `espradio_queue_recv`. Если там снова:
   - `lw a5, XXX(a5)` или `lw a5, XXX(s0)` по таблице OSI и затем `jalr a5`,  
   то при XXX = 0 в нашем слоте получится переход по нулю (pc: nil). Нужно проверить, какой это слот (offset в байтах) и заполнить его в `espradio_osi_funcs`.

4. **Поиск в build/firmware.elf.dis:**
   - Искать по тексту: `44ae4` (адрес `espradio_queue_recv`) — в связанном образе прямой ссылки может не быть (вызов идёт по регистру).
   - Искать последовательность: `3fcdf954` (g_osi_funcs_p), затем через несколько строк `116(a5)` или `74(a5)` или `5bfc` / `0747a`, затем позже `jalr a5`.
   - Или искать все `jalr a5` в диапазоне 0x4038xxxx и смотреть выше по коду: была ли загрузка из таблицы со смещением 116.

5. **Проверка стека:** Если после return из queue_recv следующая инструкция не похожа на загрузку из таблицы и jalr, а крэш всё равно с pc: 0, возможно испорчен адрес возврата (переполнение стека в нашей реализации queue_recv или в блобе). Стоит проверить размер стека wifi task и объём использования стека в `espradio_queue_recv` (println, большие буферы).

### Смещения в wifi_osi_funcs_t (для поиска по дизассему)

- 88  = 0x58 — _mutex_unlock (часто сразу после lock)
- 116 = 0x74 — **_queue_recv**
- 224 = 0xe0 — используется в mesh path
- 228 = 0xe4 — timer
- 240 = 0xf0 — timer
- 244 = 0xf4 — _wifi_reset_mac
- 264 = 0x108 — coex
- 408 = 0x198 — coex_wifi_request
- 412 = 0x19c — coex_wifi_release

После возврата из вызова по слотам 116 (queue_recv) блоб может сразу брать следующий callback из таблицы — проверить, что слот не NULL.

### Важно: wifi task в ROM

В `build/firmware.elf.dis`: `pp_create_task` (4201f5be) создаёт задачу с точкой входа **ppTask = 0x40001720**. Этот адрес в ROM — код цикла wifi task (в т.ч. вызов queue_recv и что идёт после) в нашем дизассе **не виден**. Искать «следующую инструкцию после queue_recv» в firmware.elf.dis по этой ветке нельзя; можно только:

- проверять все слоты OSI в диапазоне, который может использоваться в цикле (116 queue_recv, 120 queue_msg_waiting, и т.д.);
- минимизировать использование стека в `espradio_queue_recv` (println и большие буферы), чтобы не портить адрес возврата в ROM.

### Крэш сразу после "starting radio..."

После удаления println из queue_recv/semphr_take/give/queue_send и замены event_group_wait_bits на безопасный return крэш сместился: теперь он происходит **сразу после** вывода "starting radio...", т.е. в начале `esp_wifi_start()` (или при первом срабатывании wifi task по команде start).

**Возможные причины:**

1. **NULL в таблице OSI** — маловероятно: все слоты в `espradio_osi_funcs` явно заполнены.
2. **Внутренний указатель блоба** — блоб при start может вызывать callback/указатель, заданный не из нашей таблицы (например, из coex или другого модуля), и он остаётся 0.
3. **Переполнение стека** — стек wifi task (goroutine) мал; даже без println первый вызов в цепочке (например, semphr_take/queue_recv) может переполнить стек и стереть адрес возврата → pc: 0 при return.

**Сделано:** перед `esp_wifi_start()` вызывается `espradio_ensure_osi_ptr()`, чтобы гарантировать `g_osi_funcs_p = &espradio_osi_funcs` на случай перезаписи.

**Дальше:** если крэш сохранится — либо искать в блобе (libnet80211/libpp) вызов по указателю в начале start-пути, либо запускать wifi task на стеке большего размера (например, C-поток с заданным стеком вместо goroutine).

### Первый вызов в esp_wifi_start (слот 372)

В `libnet80211.dis`: в начале `esp_wifi_start` блоб загружает указатель из `g_osi_funcs_p`, затем по смещению **372** (0x174) загружает функцию и вызывает её с `a0=24` — это **\_wifi_malloc(24)** или **\_wifi_realloc(..., 24)**. Сделано: 1) явная запись таблицы по адресу **0x3fcdf954** в `espradio_ensure_osi_ptr()` и `espradio_prepare_memory_for_wifi()`, чтобы блоб всегда видел таблицу по ожидаемому адресу; 2) в `espradio_wifi_realloc` трактуем `ptr < 0x1000` как NULL (на случай, если блоб передаёт аргументы в другом порядке и мы бы делали memcpy из невалидного адреса).

### Вызов по смещению 200 в _do_wifi_start

После возврата из esp_wifi_start (slot 372) блоб делает tail-call в _do_wifi_start. Там по смещению **200** (0xc8) загружается указатель и делается `jr t1` — это **_dport_access_stall_other_cpu_end_wrap**. Раньше у нас там был espradio_panic → переход в обработчик исключения и pc: nil. Заменено на no-op (пустое тело), чтобы блоб продолжал выполнение.

### Диагностика: дошли ли мы до wifi_malloc

Если временно в `espradio_wifi_malloc` при `size == 24` возвращать `NULL`, блоб пойдёт по ветке ошибки и `esp_wifi_start()` вернёт код ошибки (257) вместо падения. Тогда по логу будет видно «esp_wifi_start err 257» — значит, вызов по слоту 372 до нас доходит и падение происходит позже (tail-call в _do_wifi_start или в wifi task). Если при таком изменении падение с pc: nil остаётся — значит, до wifi_malloc выполнение не доходит (например, по адресу g_osi_funcs_p блоб читает 0).

## pc: nil после queue_recv — разбор по ассемблеру (libpp.dis ppTask)

**Файл:** `raw-binary/libs/esp32c3/libpp.dis`, секция `ppTask` (строки ~19215–19300).

**Что делает блоб после возврата из queue_recv:**

1. `queue_recv(queue, sp+8, -1)` — аргумент `item = sp+8`. Мы пишем в `item` 8 байт: `[type=6, 0, 0, 0, ptr_lo, ptr_hi]`.
2. После `jalr` (возврат из queue_recv):
   - `62: lw a5,8(sp)`  — a5 = type (6).
   - `64–6e:` если type>=32 или type==13 → .L964.
   - Иначе блок 72–a0 (lock/unlock, счётчики), затем переход на .L964.
3. **.L964 (a2):**
   - `a2: lw a0,8(sp)`   — a0 = type.
   - `aa: slli a5,a0,0x2` — a5 = type*4.
   - `ae: add a5,a5,s1`  — a5 = jumptable_base + type*4 (s1 загружен из глобала в начале ppTask).
   - `b0: lw a5,0(a5)`   — a5 = jumptable[type] = адрес обработчика.
   - `b2: jr a5`         — переход по адресу обработчика.

**Важно:** перед `jr a5` блоб **не проверяет** a5 на 0. Если `jumptable[6] == 0`, выполняется `jr 0` → **pc: nil**, mcause = Illegal instruction.

**Вывод:** переход по адресу 0 согласуется с тем, что в jumptable для type 6 записан нулевой указатель (обработчик для cmd 6 не зарегистрирован или таблица не инициализирована в нашем окружении). Это не доказывает переполнение стека; переполнение стека привело бы к испорченному адресу возврата (ra) и непредсказуемому pc, а не обязательно к 0.

**Что проверить дальше:**

- Где блоб заполняет jumptable для type 6 (config_cb / ieee80211_ioctl_process) и вызывается ли этот код до первого queue_recv в ppTask.
- Отличие окружения esp-rs (где всё работает): там та же таблица заполняется раньше или из другого места (другой порядок инициализации, другой рантайм).

## Событие 41 (WIFI_EVENT_HOME_CHANNEL_CHANGE) — кто постит и как обрабатываем

**Что такое 41:** В `esp_wifi_types_generic.h` событие 41 = `WIFI_EVENT_HOME_CHANNEL_CHANGE` («смена home channel», не при скане). Данные события: до 16 байт (структура с полями канала и т.п., см. `wifi_event_ap_staconnected_t` и аргументы для HOME_CHANNEL).

**Кто постит:** В декомпе `firmware.elf.c`:
- `chm_set_home_channel` (стр. ~30287): при смене home channel вызывает `wifi_event_post(0x29, local_30, 0x10)` — то есть id=0x29 (41), буфер `local_30` (старый/новый канал), размер 0x10.
- `wifi_event_post` (стр. 11439) делает вызов слота OSI 0xb4: `(**_g_osi_funcs_p + 0xb4)("WIFI_EVENT", param_1, param_2, param_3, 0xffffffff)` — т.е. наш `_event_post` с base="WIFI_EVENT", event_id=41, event_data=указатель на 16 байт, event_data_size=0x10.

**Цепочка до падения (по логу):**  
coex_wifi_release(36) → (в блобе дальше по коду) → в какой-то момент вызывается `chm_set_home_channel` → `wifi_event_post(41, data, 16)` → наш `espradio_event_post("WIFI_EVENT", 41, data, 16, 0xffffffff)` → мы делаем printf, кладём событие в очередь, возвращаем 0 → управление в блоб: `wifi_event_post` возвращается, затем `chm_set_home_channel` пишет `DAT_ram_3fc8186c = 1` и делает `return`. Дальше выполнение идёт в **вызывающий** `chm_set_home_channel` (например, `chm_change_channel` или код после `chm_end_op`). Падение pc: nil происходит уже **после** возврата из нашего `event_post`, т.е. в коде блоба (или в следующем вызове OSI).

**Наша реализация (сверка с контрактом):**
- Мы возвращаем 0 — блоб не идёт по ветке ошибки (как в IDF).
- Очередь: один список событий, lock, при post добавляем в хвост.
- При `event_data_size > 0 && event_data` мы делаем `malloc` + `memcpy` для `e->data` — для 41 с size=0x10 данные копируются.
- `run_once` вызывается из Go (radio.go) в цикле; за один вызов обрабатывается одно событие из головы списка, вызываются все зарегистрированные обработчики по base+id (в т.ч. `ESP_EVENT_ANY_ID`), с проверкой `h->handler` перед вызовом.
- Дефолтный обработчик: `espradio_event_register_default_cb()` вешает `espradio_dummy_event_cb` на `WIFI_EVENT` + `ESP_EVENT_ANY_ID` — для 41 он просто вызывается и ничего не делает (логирует при ESPRADIO_OSI_DEBUG).

**Отличие от esp-rs:** В esp-rs используется либо полный event loop IDF (`esp_event_loop_run` в отдельной задаче), либо обёртка поверх него; `_event_post` из адаптера OSI вызывает нативный `esp_event_post`. События обрабатываются в контексте задачи event loop, а не в контексте вызова из wifi task. У нас обработка отложенная (run_once из Go), но контракт для блоба тот же: post только ставит событие в очередь и возвращает 0. Имеет смысл проверить в esp-rs: 1) какой слот OSI используется для _event_post (0xb4) и что именно передаётся; 2) вызывается ли у них что-то ещё сразу после возврата из post в пути chm_set_home_channel (таймеры, другие osi-вызовы), чего у нас может не быть или работать иначе.

## Причина pc:nil после event_post id=41: колбэк _g_chm+88

**Цепочка по дизассемблеру (libnet80211.dis):**

1. После возврата из `event_post` управление в `chm_set_home_channel`: запись `DAT_ram_3fc8186c = 1`, затем `return`.
2. Вызывающий — `chm_init`: следующий вызов — `chm_set_current_channel(&uStack_34)`.
3. В конце `chm_set_current_channel`: загрузка `t1 = *(_g_chm + 88)` (инстр. `lw t1, 88(a5)`), затем `jr t1` — переход по этому указателю.
4. Если по смещению 88 в контексте канального менеджера (`gChmCxt`) не записан указатель на функцию, там 0 → переход по нулю → **pc: nil**.

**Вывод:** Блоб ожидает по смещению 88 в `gChmCxt` колбэк «phy change done» (или аналог). В нашем окружении это поле не инициализируется (в IDF его, видимо, выставляет другой слой при инициализации).

**Исправление (radio.c):** Перед `esp_wifi_init_internal()` вызываем `espradio_set_chm_phy_done_cb()`: объявлен `extern char gChmCxt[252]`, по смещению 88 записывается указатель на no-op `espradio_chm_phy_done_noop(void)`. После этого `chm_set_current_channel` в конце делает `jr` в нашу заглушку вместо перехода по нулю.
